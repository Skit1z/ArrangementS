## 第六波：修复在岗人员时区 bug (commit 42b23dc)

### 根因
`datetime.now(BEIJING_TZ)` = `datetime.now(timezone(timedelta(hours=8)))`
这个写法有陷阱：它只是把系统当前时间加上 +08:00 标签，**并不转换时间值**。
- 服务器 UTC 时：贴了 +08:00 标签的 UTC 时间值 → 偏移 8h
- 服务器北京时间时：刚好正确

### 修复
```python
now_utc = datetime.now(timezone.utc)           # UTC 时间
now_beijing = (now_utc + timedelta(hours=8)).replace(tzinfo=BEIJING_TZ)  # → 北京时间
```
先做数学转换（+8h）再贴标签，保证时间值正确。

```
TypeError: unsupported operand type(s) for +=: 'NoneType' and 'int'
```

### 在岗人员不显示 根因 (commit a8f0740)
两个 bug：
1. `_adjacent_shift_people` 用 `db.execute(select(PersonProfile))` 返回 Row 对象
   访问 `p.full_name` 抛 `KeyError: 'full_name'`
   → 改用 `db.scalars()` 显式返回 ORM 实体
2. SQL 端时区过滤不可靠（PG 隐式转换行为不确定）
   → 还原为拉取所有已发布记录后 Python 端做范围比较

### 关键教训
- `db.execute(select(Entity))` vs `db.scalars(select(Entity))`：
  - 测试环境下两者可能都返回 Entity
  - 生产 Cython 优化下 `db.execute` 可能返回 Row
  - 要 Entity 对象请用 `db.scalars()`
- SQL 端时区过滤 vs Python 端过滤：
  - Python 端更可控、可调试、不依赖 DB 默认时区设置
  - 性能差异可以忽略（拉少量数据后过滤）

## 第九波：后端 CI 提速（方案1+2，未 commit）

### 问题
后端 CI 跑满约 2min，主因是排班求解器 CP-SAT 每次 `generate()` 吃满 `max_time_seconds=15.0` 上限，且 `num_search_workers=1`（单线程保可复现）。
- `test_reproducible_generate`：连跑 2 次 generate → 30.5s
- `test_add_task_to_published_plan...`：1 次 → 12.9s
- pytest 总时长 ~65s（本地）

### 修复
- 方案1：把 `max_time_seconds` 透传进 `generate()`（`schedule_service.py`）→ `build_solver_input()`（`eligibility.py`）。
  - `generate(... max_time_seconds: float | None = None)`，None 时回落到 SolverInput 默认 15.0 → **生产路径不变**。
  - 所有测试 `generate(...)` 调用加 `max_time_seconds=2.0`。
- 方案2：`test_reproducible_generate` 第二次完整 `generate()` 换成「同 seed 直接复跑 `solve(build_solver_input(...))`」，省掉一次完整管线重建，仍验证确定性。
  - 注意 `solve()` 返回的 `assignments` 值是 person_id 字符串（非 Person 对象），比较时直接 `str(p)`。

### 结果（本地）
- pytest：65s → 26s
- `test_reproducible_generate`：30.5s → 4.5s；`test_add_task_to_published_plan...`：12.9s → 2.1s
- ruff format/check 仍全绿，194 passed
- 生产 API（`app/api/v1/schedule.py:60`）未传该参数，保持 15s 不变
